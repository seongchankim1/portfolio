트러블슈팅

쿼리 최적화 및 데이터 접근 개선

문제점: 복잡한 조인 및 연관관계, N+1 문제로 인한 대용량 데이터 조회 성능 저하

대안 목록:

  • 단순 Join – 불필요한 전체 스캔 발생
  • 복합 인덱스 + QueryDSL – 명확한 인덱스 적용
  • Query Hint – 특정 인덱스 강제 사용, 유지보수 복잡

선택: 복합 인덱스 추가 및 QueryDSL 활용

구현:

@Table(name = "users", indexes = {
  @Index(name = "idx_email", columnList = "email"),
  @Index(name = "idx_nickname", columnList = "nickname")
})
public class User extends TimeStamp {}
@Override
public List findByProjectIdAndUserIds(Long projectId, List userIds) {
  return queryFactory
    .selectFrom(authority)
    .join(authority.user).fetchJoin()
    .where(
      authority.project.id.eq(projectId),
      authority.user.id.in(userIds)
    )
    .fetch();
}

결과: 1,000만 건 이상 조회 시에도 2~3ms 응답 속도 유지

데이터 파티셔닝 및 효율적 관리

문제점: 로그 데이터 누적으로 인한 전체 테이블 스캔 성능 저하

대안 목록:

  • 단순 인덱스 튜닝 – 장기적 한계
  • 캐싱 – 캐시 관리 복잡도 증가
  • MySQL 파티셔닝: 월별 RANGE 파티셔닝 도입

구현:

CREATE TABLE activity_log (
  user_id BIGINT,
  created_at DATETIME NOT NULL,
  action_type VARCHAR(50),
  action_detail TEXT,
  PRIMARY KEY (user_id, created_at)
)
PARTITION BY RANGE (YEAR(created_at) * 100 + MONTH(created_at)) (
  PARTITION p202502 VALUES LESS THAN (202502),
  PARTITION p202503 VALUES LESS THAN (202503),
  PARTITION pmax VALUES LESS THAN MAXVALUE
);

결과: 파티션 프루닝 자동 적용으로 조회 성능 50% 이상 향상 및 대량 데이터 삭제 0.1초 처리
